# Remote Agent 编译与打包指南

本指南详细说明了 Remote Agent 系统的编译逻辑、打包流程以及如何在本地或目标服务器上进行部署。系统集成了 React 前端、Go 后端代理服务、Go 消息桥接守护进程以及 C 语言原生的终端服务器，我们通过根目录下的统一 `Makefile` 实现了高度自动化、规范化的构建流程。

---

## 1. 编译前置依赖 (Prerequisites)

在开始编译之前，请确保您的编译环境已安装以下工具链和依赖库：

### 通用依赖
- **Git**: 用于解析版本标签和 Commit Hash。
- **Go (>= 1.22)**: 用于编译 `remote-agents` 和 `cc-connect` Go 服务。
- **Node.js (>= 18)** 和 **Yarn (3.6.3)**: 用于编译前端静态资源（前端采用 Yarn 3.6.3 并配置了 `nodeLinker: node-modules`）。
- **CMake (>= 3.12)**: 用于配置 native `ttyd` C 语言项目的编译。

### 系统特有开发库（编译 C 终端服务器 `ttyd` 所需）
- **macOS**:
  ```bash
  brew install cmake libuv json-c openssl libwebsockets
  ```
- **Linux (Debian/Ubuntu)**:
  ```bash
  sudo apt-get update
  sudo apt-get install -y cmake build-essential libuv1-dev libjson-c-dev libssl-dev libwebsockets-dev autoconf automake libtool
  ```

## 2. 统一构建命令 (Root Makefile Targets)

我们在项目根目录下提供了一套统一的 `Makefile` 指令，用于调度所有子模块的构建：

| 构建指令 | 执行逻辑与说明 |
| :--- | :--- |
| `make help` | 显示当前编译机的主机名、操作系统、处理器架构，并列出所有可用指令。 |
| `make all` | **一键编译全部**。依次构建前端、ttyd 终端服务器、cc-connect 守护进程和 Go 代理服务器。 |
| `make frontend` | **构建前端资源**。执行 `yarn install` 和 `yarn build` 生成静态网页，并通过 Gzip 压缩后生成 C 语言字节头文件 `src/html.h`。 |
| `make ttyd` | **构建 C 终端服务器**。自动识别系统，执行 CMake 并在 `build-ttyd/` 目录中编译，最后将可执行文件复制到 `build/ttyd`。 |
| `make cc-connect` | **构建消息桥接进程**。自动编译 Web 后台资产并编译 Go 二进制文件，最后将可执行文件复制到 `build/cc-connect`。 |
| `make cc-connect-noweb` | **快速构建桥接进程**。不重新编译 web 后台资产，直接编译 Go 二进制，适合日常频繁的 Go 代码调试。 |
| `make agent` | **构建代理主服务**。编译 Go 服务端主程序，将可执行文件输出到 `build/remote-agents`。 |
| `make package` | **一键打包**。自动执行完整编译后，将全部二进制及前端资产打包压缩为目标机器专有的 `.tar.gz` 归档包。 |
| `make clean` | **清理环境**。彻底清除所有生成的二进制文件、编译缓存目录 (`build/`, `build-ttyd/`, `dist/`) 以及中间件文件。 |

## 3. 产物集中管理设计 (Centralized Output)

为了使项目结构保持绝对整洁，避免传统编译方式下可执行文件散落的问题，所有的构建产物在编译完成后，都会**统一集中**输出到项目根目录下的 **`build/`** 目录中：

```
remote-agents/
├── build/                 # 所有编译可执行文件的集中存放目录
│   ├── remote-agents      # Go 后端代理服务主程序
│   ├── ttyd               # Native C 语言终端服务器
│   └── cc-connect         # Go 消息桥接守护进程
```
> [!TIP]
> 根目录下的 `build/`、`build-ttyd/` 和 `dist/` 均已被写入项目根目录的 `.gitignore` 中，开发调试时无需担心编译产物会被意外提交。

## 4. 主机名与平台差异化版本哲学 (Environment Distinction)

在分布式或多设备部署场景中，直接使用同一次 Git Commit 的哈希值会导致编译出的 Linux 和 Mac 二进制文件在运行中难以区分。

为了解决这个问题，Remote Agent 引入了**编译机硬件元数据注入**机制：

1. **自动采集元数据**: 编译时，构建系统会自动采集编译主机的 **系统平台 (`OS`)**、**处理器架构 (`Arch`)** 以及 **网络主机名 (`Hostname`)**。
2. **多语言元数据注入**:
   - **Go 组件 (`remote-agents`, `cc-connect`)**: 通过 `go build -ldflags` 将元数据（如 `v20260526-darwin-arm64-MacBook.local`）作为硬编码字符串注入到主进程的 `commit` 变量中。
   - **C 组件 (`ttyd`)**: 通过 CMake 动态获取主机信息，并将其注入为 `-DTTYD_VERSION` 预编译宏。
3. **版本查看验证**:
   运行任意编译产物并带上 `--version` 或 `-version` 选项，即可瞬间验证其详细构建环境信息：
   - **Go 代理服务**:
     ```bash
     $ ./build/remote-agents -version
     remote-agents v20260526-2-24-g9af36ba-dirty
     commit:  9af36ba-darwin-arm64-MacBook-Air-8.local
     built:   2026-05-28T08:27:45Z
     ```
   - **终端服务**:
     ```bash
     $ ./build-ttyd/ttyd --version
     ttyd version 1.7.7-9af36ba-Darwin-arm64-MacBook-Air-8.local
     ```

## 5. 本地打包与目标服务器部署 (Packaging & Deployment)

当您需要将编译好的服务部署到目标服务器时，请采用 **`make package`** 工作流：

### 第一步：一键构建安装包
在终端执行：
```bash
make package
```
打包系统会在 `dist/` 目录下生成与当前构建环境强绑定的压缩包，如：
`dist/remote-agents-v20260526-2-24-g9af36ba-dirty-darwin-arm64-MacBook-Air-8.local.tar.gz`

### 第二步：解压部署
将上述生成的压缩包拷贝到目标服务器，并执行解压：
```bash
# 解压到当前文件夹
tar -zxvf remote-agents-*.tar.gz
```

解压后的标准部署目录结构如下：
```
remote-agents-[version]-[os]-[arch]-[hostname]/
├── bin/
│   ├── remote-agents      # 主系统服务二进制
│   ├── ttyd               # 终端服务器二进制
│   └── cc-connect         # 桥接服务二进制
└── dist/                  # 前端静态 HTML 资产目录
```

### 第三步：运行服务
进入解压后的文件夹，通过以下指令启动 Remote Agent 服务：
```bash
# 启动代理服务并指定刚编译好的本地 ttyd 二进制路径
./bin/remote-agents --ttyd-bin ./bin/ttyd --static ./dist
```
